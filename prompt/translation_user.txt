Recent compiler feedback from this project:
- `shiftR` is not in scope in generated Cryptol
- `(&)` is not in scope in generated Cryptol
- `fromIntegral` is not in scope in generated Cryptol
- `(n : Integer, lgw : Integer) -> Integer` caused a parse error
- Using a type parameter `n` as a runtime value caused `Expected a value named 'n' but found a type instead`. Fix: use backtick `` `n `` to demote type to value
- Using `[Integer][8]` style sequence types is invalid because the length must be a numeric type expression, not the value type `Integer`
- `[0 .. `n - 1]` caused `The boundaries of .. sequences should be valid numeric types`. Fix: use `take`{n} [0 ...]` or type-level bounds `[0 .. (n-1)]` (no backtick)
- Mixing bitvector widths (e.g., `[7]` + `[9]`) caused `Type mismatch: Expected type 9, Inferred type 7`. Fix: use `zext` to widen the narrower value
- `div` and `mod` (also `/` and `%`) are valid Cryptol operators and should be used freely for modular arithmetic
