该项目最近的编译器反馈：
- 生成的 Cryptol 中 `shiftR` 不在作用域内
- 生成的 Cryptol 中 `(&)` 不在作用域内
- 生成的 Cryptol 中 `fromIntegral` 不在作用域内
- `(n : Integer, lgw : Integer) -> Integer` 会导致解析错误
- 把类型参数 `n` 当作运行时值会导致 `Expected a value named 'n' but found a type instead`。修复方式：使用反引号 `` `n `` 将类型降格为值
- `[Integer][8]` 这种序列类型写法是非法的，因为长度必须是数值类型表达式，而不能是值类型 `Integer`
- `[0 .. `n - 1]` 会导致 `The boundaries of .. sequences should be valid numeric types`。修复方式：使用 `take`{n} [0 ...]` 或类型级边界 `[0 .. (n-1)]`（不要用反引号）
- 混用不同位宽的比特向量（例如 `[7]` + `[9]`）会导致 `Type mismatch: Expected type 9, Inferred type 7`。修复方式：使用 `zext` 扩展较窄的值
- `div` 和 `mod`（以及 `/` 和 `%`）是合法的 Cryptol 运算符，应自由用于模运算
- 不要猜测 Cryptol 中存在 `BytesToBits`、`BitsToBytes`、`zeroExtend` 之类的名字；如果证据中没有明确给出，就不要直接调用
- 不要默认把 `^` 当作整数幂；先确认该运算符在当前操作数类型和当前上下文中成立
- 不要把 `zext` 当成 Bit 直接转 Integer 的万能转换；涉及 Bit、位向量和 Integer 的混合算术时，应先统一域
- 不要把值级 `len`、`numBlocks`、`startCount` 之类的变量直接用于 `take len [0 ...]`、`[0 .. (len - 1)]` 这类有限序列生成
- 如果循环控制量来自运行时/值级表达式，应在“完全静态展开”和“值级递归”之间二选一，不要混用
- 不要把数组状态和计数器之类不同类型字段塞进同一个序列；应使用 tuple 或 record
- 对 `[256]` 数组的索引相关变量，位宽设计应覆盖整条链：`len/start/j/j+len/start+2*len`，不要只补某一个函数签名
- 对算法核心路径上的 helper（例如 NTT 的 zeta 查表/幂表），不要一边保留 `undefined` 一边继续生成看似完整的主实现；若证据中已有预计算表或查表模式，应优先直接采用
- 推导式生成器变量只在该推导式内部有效；不要把像 `i <- ...` 这样的绑定变量拿到并列 `where` 定义里使用，否则会触发 `Value not in scope`
- 反引号只用于数字类型参数，不要对普通值级变量或推导式绑定变量使用反引号
- 不要把类型参数 `{j}`、`{n}` 当成运行时递归计数器来写 `if`、减一或递归推进
- 如果参数参考只能把某个参数解析到参数族/上下文，而不能唯一确定具体值，就不要继续假装它足以支撑完整可执行实现；必要时应保守降级
- 注释里写“依赖模块已提供某 primitive/type”并不等于依赖已经真的存在；只有用户明确给出的依赖代码、当前模块定义或实际 import 才算满足依赖

翻译时请额外遵守：
1. 若某个操作名只出现在伪代码里，但没有在依赖代码、RAG 片段或当前模块中被明确给出，则不要直接调用同名或猜测名。
2. 对涉及量化数字参数、位提取和模运算的表达式，优先拆成几个带显式类型的小 helper，而不是写成一行混合推导式。
3. 先区分清楚类型级参数和值级数字，再决定是否使用反引号降格。
4. 若证据不足以支持完整可执行实现，优先输出最小、合法、保守的骨架，并用简短的 `// UNRESOLVED:` 注释说明。
5. 对循环密集型算法，写代码前先决定是“完全静态展开”还是“值级递归”；不要在一个函数里同时使用动态递归控制和依赖值级变量长度的有限序列生成。
6. 如果状态中混合了数组、计数器、层号、标志等不同类型字段，统一用 tuple 或 record 传递。
7. 如果算法依赖核心常量表、查表或预计算表，而证据里已经提供了它们，优先直接采用这些表，不要改写成证据不足的 `undefined` helper。
8. 如果在推导式里绑定了变量，而某个中间量依赖这个变量，请把该中间量内联到推导式元素里，或显式作为 helper 参数传递；不要默认它能在并列 `where` 中可见。
9. 如果某段逻辑需要递归推进计数，请使用显式值级计数器；如果某段逻辑是静态类型级构造，就保持它为真正的类型级构造；不要把两者混在同一递归里。
10. 如果某个关键外部 primitive/type 并未被当前证据真正提供，不要围绕它继续生成伪完整主逻辑；应改用现有依赖，或回退为保守骨架。
