# Agent4Cryptol: A Multi-Agent Framework for Translating Standard Algorithm Specifications into Cryptol



## Abstract

摘要建议写成 4 段式信息：

1. **问题**：标准算法文档到 Cryptol 的人工建模成本高、要求强语法/类型精度，通用 LLM 直接生成容易出错。
2. **方法**：提出 Agent4Cryptol，多智能体闭环框架，包含 Spec2IR Agent、IR2Cryptol Agent、Compilation-Fix Agent。
3. **核心机制**：引入中间表示 IR、Cryptol 专用 RAG、提示词约束、编译反馈修复。
4. **结果**：在标准文档数据集上验证，提升编译成功率、减少修复轮次、优于 baseline。

------

# I. Introduction

### 1.1 Problem Background

- 形式化密码算法建模很重要，尤其是 Cryptol 这种面向规格与验证的语言。
- 标准文档中的算法通常是自然语言 + 伪代码 + 散落参数，直接转成 Cryptol 很难。
- 通用 LLM 擅长主流语言，但对 Cryptol 这种小众、强约束、类型敏感语言不稳定。

### 1.2 Motivation

- 人工从标准文档写 Cryptol 成本高、周期长、容易遗漏隐藏参数和类型约束。
- 直接 doc-to-Cryptol 会同时面对：
  - 文档解析问题
  - 算法结构理解问题
  - Cryptol 语法/类型问题
- 所以需要一个**分阶段、多智能体、可闭环修复**的框架。

### 1.3 Key Idea

- 不直接一步到位生成 Cryptol。
- 先把标准文档结构化成 **IR JSON**。
- 再从 IR 结合 RAG 和提示词生成 Cryptol。
- 最后通过编译器反馈迭代修复，直到得到可编译结果。

### 1.4 Contributions

1. 提出 **Agent4Cryptol**，一个面向标准算法文档到 Cryptol 的多智能体闭环生成框架。
2. 设计一个适合标准算法文档解析与建模的 **中间表示 IR**，连接文档理解与形式化代码生成。
3. 提出 **Compiler-in-the-loop repair** 机制，利用 Cryptol 编译反馈进行自动修复。
4. 构建一个用于评估标准文档到 Cryptol 的数据集/benchmark，并从有效性、效率、消融和案例多个角度进行评测。

# II. Background

## 2.1 Cryptol and Formal Modeling

- Cryptol 的定位：规格描述、位级建模、形式化验证前端。
- Cryptol 的特点：
  - 强类型
  - 位向量/序列运算丰富
  - 模块和函数定义规范严格
  - 对语法与类型错误敏感

## 2.2 Standard Algorithm Documents

- 标准文档的组成：
  - 算法框/伪代码
  - 文本描述
  - 参数表
  - 上下文约束
- 难点：
  - 输入输出未显式标注
  - 部分参数隐藏在正文
  - 算法依赖跨页或跨章节
  - 同一算法在不同 parameter set 下实例化不同

## 2.3 LLM-based Multi-Agent Systems

- 简要介绍 agent 思路：不同 agent 负责不同子任务。
- 为什么不用单 agent：你的任务天然是**文档理解 → 结构化 → 代码生成 → 修复**四类不同问题。
- 可简单提 LangGraph / tool-using workflow。

## 2.4 Compiler Feedback and Repair

- 代码生成的终点不能只是“看起来像”，而要“能加载/编译”。
- 编译器反馈能提供可靠约束。
- 对 Cryptol 这种形式化语言，编译器反馈修复比纯自然语言自反思更有效。

# III. Agent4Cryptol Methodology

## 3.1 Overall Framework

给总流程图，说明：

**Input:** standard algorithm documents
**Output:** compilable Cryptol code

整体流程：

**Standard Document → Spec2IR Agent → IR JSON → IR2Cryptol Agent → draft Cryptol → Compilation-Fix Agent → final Cryptol**

这里建议明确说明：

- 框架是**串行闭环**
- 前两步是“理解与生成”
- 最后一步是“验证与修复”
- 修复失败超过阈值则终止

## 3.2 Intermediate Representation Design

写清楚 IR 的作用：

- 降低文档噪声
- 统一算法结构
- 为代码生成提供稳定输入
- 便于后续检查、索引构建

建议写 IR 字段设计

## 3.3 Spec2IR Agent

说明它负责什么：从标准算法文档中抽取中间语言IR

### 输入

- PDF/标准文档
- 算法框、章节文本、参数描述

### 输出

- 单函数 JSON
- function index
- parameter sets
- 依赖关系（可选）

### 核心子任务

- 算法定位
- 伪代码块提取
- 输入输出识别
- 隐含参数抽取
- 页面与标签对齐
- 函数依赖识别
- schema 校验

### 难点

- 正文参数不显式
- 同名参数跨算法复用
- 不同 parameter set 下的实例化

## 3.4 IR2Cryptol Agent

说明它是“从结构化 IR 到 Cryptol”的生成 agent。

### 输入

- 单函数 IR JSON
- parameter_sets.json
- RAG 检索结果
- 提示词模板

### 输出

- 初版 Cryptol 代码

### 核心机制

- 基于 IR 的受控生成
- Cryptol 专用 RAG
- few-shot / one-shot 示例
- 语法约束提示
- 类型/模块命名约束

### 重点强调

- 为什么 IR 能比原始文档更利于生成
- 为什么需要专门的 Cryptol 规则知识库，而不是通用代码语料

## 3.5 Compilation-Fix Agent

### 输入

- 初版 Cryptol
- 编译器报错
- 上一轮修复历史
- 原始 IR

### 输出

- 修复后的 Cryptol

### 流程

1. 调用 Cryptol 编译/加载
2. 收集错误信息
3. 分类错误
   - 语法错误
   - 作用域错误
   - 类型错误
   - import / module 错误
   - 运算符误用
4. 生成 patch
5. 回写新代码
6. 重新编译

### 关键设计

- 错误消息结构化
- patch-based repair
- 保留历史上下文，防止来回震荡
- 设置最大迭代轮数

## 3.6 Implementation Details

- agent orchestration 框架
- RAG 库构成
- 编译器工具接口
- 日志与中间产物保存
- 终止条件与失败恢复

# V. Results and Analysis

## 5.1 RQ1: End-to-End Effectiveness

展示： 直接放一张表格，列出 B1 到 B5 在 Easy/Medium/Hard 数据集上的 Compile Success Rate。

- **核心论点：**B5 (Full Agent4Cryptol) 实现了正确率的显著上升，证明了多阶段、多智能体闭环设计的整体有效性。

## 5.2 RQ2 & RQ3: Agent-level Ablation Study

- **The Power of IR (B1 vs B3):** 解释为什么B3 (Spec2IR + 基础 Prompting) 优于B1。重点分析标准文档里的隐式参数和排版噪声是如何让B1准确率低，而 IR 又是如何通过结构化提供稳定上下文的。

- **The Power of Specialized Generation (B3 vs B4):** 解释 B4 (Spec2IR + IR2Cryptol) 为什么优于B3。这里的论点是：有了好的（IR），还需要专门具备 Cryptol 领域知识、提示词约束的 IR2Cryptol Agent，才能减少 Haskell 幻觉。(原本的 RAG、Few-shot 作为解释 B4 为什么强的内部原因）

- **The Power of Compiler Feedback (B4 vs B5, 以及 B1 vs B2):** 分析 Compilation-Fix 的兜底作用。展示加上 Fix Agent 后，那些因为拼写错误、类型未对齐导致的差错是如何被救回来的。

## 5.3 RQ3: Efficiency & Iteration Analysis (效率与修复迭代分析)

- 这里重点分析 B2 和 B5 在调用 Fix Agent 时的表现差异。
- **核心论点：** 虽然 B2 (Prompting + Fix) 也能修代码，但因为它的Prompting太差，导致修复轮次极高，甚至经常修不好。而B5 因为前面有 Spec2IR 和 IR2Cryptol ，只需很少的修复轮次就能收敛。这证明了**多智能体的协作是极大提升了系统的运行效率**。

# VI. Discussion

## 6.1 Why IR Matters

- 为什么直接 doc-to-code 不稳
- 为什么 IR 提高了可控性与可解释性

## 6.2 Common Error Patterns

总结常见错误：主要是容易产生Haskell编程语言幻觉。

- import 错误
- 运算符误用
- 类型不匹配
- 隐含参数遗漏
- 模块结构错误

## 6.3 Limitations

- 对于存在算法内部函数依赖关系的算法错误率较高，主要是会存在代码错误传递（比如第一个子函数错了，那么后面需要这个子函数的函数肯定是不对的）当前框架以自底向上或独立的单函数生成为主，对于深层嵌套的函数调用链，底层函数的逻辑幻觉容易向上传递（Error Cascading）。未来的研究可以考虑引入**全局依赖图（Dependency Graph）约束**或自顶向下的接口契约（Interface Contract）生成机制来缓解这一问题。
- 对于依赖外部函数的函数目前只能支持形式上的建模，但是实际上不能编译运行（Cryptol语言不像其它编程语言那样有库函数，有的时候需要依赖其它外部函数，大部分情况下都是hash函数）-- 对于依赖外部复杂密码学原语（如 SHA-3）的算法，本框架目前采用**接口抽象声明（Abstract Interface Declaration）**的方式生成类型签名的 Cryptol 代码，以确保当前模块的类型检查和语法编译通过。完整的跨模块联合验证留待未来工作探索。
- 本质上对于最终的代码验证，应该是算法标准提出方，提供的正确地测试向量，但是算法完整的Cryptol建模基本上很难完全建模成功，所以用不到这个。这是致命问题。 --- 本文的评估指标（编译成功率）主要衡量了代码在语法、类型和作用域层面的正确性（Type-check & Compilation Pass）。由于密码学标准算法的完整实例通常非常庞大，且部分底层组件采用抽象接口，目前框架尚未实现基于官方测试向量的**全系统端到端动态语义验证（Dynamic Semantic Verification）**。在未来的工作中，我们计划引入 Cryptol 的测试框架以及 SAW（Software Analysis Workbench），将 Agent4Cryptol 生成的模块与 C/Rust 实现进行等价性证明，从而实现从语法正确到语义正确的跨越。
